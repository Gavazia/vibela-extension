# Delta for Extension Shell

## ADDED Requirements

### Requirement: Chrome MV3 WXT shell

The extension SHALL be a Chrome MV3 extension generated by WXT, and the Chrome unpacked build MUST load without manifest errors.

#### Scenario: Load unpacked build in Chrome

- GIVEN a completed WXT Chrome build output
- WHEN the user loads the output directory through `chrome://extensions` in developer mode
- THEN Chrome MUST install the extension without manifest errors
- AND the VibeCopilot toolbar action MUST be visible.

### Requirement: Per-tab toolbar overlay toggle

The toolbar action MUST toggle the overlay active state for only the current tab, and MUST NOT change overlay state in other tabs.

#### Scenario: Toggle one tab independently

- GIVEN two allowed web pages are open in separate tabs
- WHEN the user clicks the toolbar action while the first tab is active
- THEN the overlay MUST become active in the first tab
- AND the overlay MUST remain inactive in the second tab.

#### Scenario: Toggle off in the same tab

- GIVEN the overlay is active in an allowed tab
- WHEN the user clicks the toolbar action again for that tab
- THEN the overlay MUST become inactive for that tab
- AND the content script MUST remove or hide interactive overlay UI for that tab.

### Requirement: Restricted schemes fail quietly

The extension MUST detect restricted or unsupported URL schemes, including `chrome://`, `chrome-extension://`, `edge://`, `about:`, `moz-extension://`, `devtools://`, and `view-source:`, and MUST fail quietly without injecting host-page UI.

#### Scenario: Restricted Chrome page

- GIVEN the active tab URL uses a restricted scheme such as `chrome://extensions`
- WHEN the user invokes the toolbar action
- THEN the extension MUST NOT create a Shadow DOM host in the page
- AND the user MUST NOT see an unhandled exception or broken overlay.

### Requirement: Closed Shadow DOM React mount

The content script MUST mount the React overlay UI inside an extension-owned closed Shadow DOM host, and page-level CSS MUST NOT style overlay internals.

#### Scenario: Overlay mounts in isolation

- GIVEN an allowed web page with global CSS that targets common element names and classes
- WHEN the overlay is activated
- THEN the content script MUST create exactly one extension-owned host element
- AND the React UI MUST render inside a closed Shadow DOM root attached to that host
- AND host-page CSS MUST NOT visibly alter the overlay controls.

### Requirement: Overlay lifecycle across navigation

The overlay MUST preserve expected lifecycle behavior across SPA route changes and full navigations.

#### Scenario: SPA route change keeps overlay mounted

- GIVEN the overlay is active on an allowed single-page application
- WHEN the page changes route without a full reload
- THEN the overlay MUST remain mounted
- AND prompt metadata SHOULD reflect the current path after the route change.

#### Scenario: Full navigation remounts safely

- GIVEN the overlay was active in a tab
- WHEN the tab performs a full page navigation to another allowed URL
- THEN the old page overlay MUST NOT leave orphaned UI behind
- AND the new page content script MUST initialize safely without duplicate hosts.

### Requirement: Browser storage ownership

The extension MUST use `chrome.storage.session` for the per-tab active flag, `chrome.storage.local` for annotation drafts, and `chrome.storage.sync` for user preferences.

#### Scenario: Persist annotation draft and preference separately

- GIVEN the user activates the overlay, saves an annotation, and changes a preference
- WHEN storage is inspected through Chrome extension DevTools
- THEN the current tab active flag MUST be owned by session storage
- AND the annotation draft MUST be owned by local storage
- AND the preference MUST be owned by sync storage.
